## 📌 1/8 작업 정리 — 예외처리 설계 및 구현

거의 막바지에 달했기 때문에 **행동·입력 실수·네트워크 상황에서 발생할 수 있는 예외를 명확히 정의하고 처리하는** 작업을 했다.  

---

### 1. 회원가입 예외처리

회원가입 과정에서 실제로 가장 빈번하게 발생할 수 있는 실패 케이스를 기준으로 예외를 나눴다.

#### ① 필수값 누락
- 이메일 또는 비밀번호가 비어 있는 상태에서 회원가입 시도
- API 요청 자체를 보내지 않고 즉시 차단
- “입력값이 부족하다”는 점을 명확히 안내

→ 불필요한 서버 요청 방지

---

#### ② 입력 형식 / 정책 위반
- 이메일 형식이 올바르지 않은 경우
- 비밀번호 정책(최소 길이, 특수문자 포함 등)을 만족하지 못한 경우

→ 단순히 “실패”가 아니라 어떤 규칙을 만족하지 못했는지 명시적으로 안내

---

#### ③ 서버 검증 실패
- 프론트엔드에서는 입력 형식과 필수값만 검증
- 실제 가입 가능여부(이미 가입된 이메일 등)는 서버에서 판단

→ 역할을 분리!


---

### 2. 입력값 누락에 대한 공통 예외처리 (Todo / Category)

Todo 및 Category 관련 기능 전반에서  
**“제목/이름 없이 저장·수정하는 상황”**을 공통 예외로 정의했다.

- 할 일 생성 시 제목 미입력
- 할 일 수정 시 제목 미입력
- 카테고리 생성 시 이름 미입력
- 카테고리 수정 시 이름 미입력

이 경우:

- API 요청을 보내지 않음
- 즉시 사용자에게 안내 메시지 표시

→ 데이터 무결성 훼손을 프론트 단계에서 차단

---

### 3. 상황별 모달 설계 (여러 경우의 수 대응)

모달은 단순 오류 표시가 아니라  
**현재 상황을 설명하고, 사용자의 다음 행동을 유도하는 장치**로 사용했다.

모달로 처리한 주요 상황:

- 필수 입력값 누락
- 입력 형식/정책 위반
- 로그인 실패
- 회원가입 실패
- 회원가입 성공 후 로그인 유도
- 서버 에러 또는 인증 관련 실패

각 상황마다:

- 제목과 메시지를 분리
- 같은 모달 컴포넌트를 상황에 따라 재사용

→ UI는 단순하게 유지하면서 의미만 달라지도록 설계

---

### 4. 중복 클릭 / 로딩 상태 처리 (실제 적용된 범위)

로그인 및 회원가입 요청에서:

- `loading` 상태를 두고
- 요청이 진행 중일 때 동일 함수 재실행을 차단

이를 통해:

- 버튼 연타로 인한 중복 API 호출 방지
- 사용자에게 “처리 중” 상태를 명확히 전달

※ 해당 처리는 **로그인 / 회원가입 흐름에 한해 적용**

---

### 5. Enter 키 처리 (사용자 UX 개선)

로그인 및 회원가입 화면을 `<form>` 기반으로 구성하여:

- 버튼 클릭
- Enter 키 입력

이 두 경우가 **완전히 동일한 submit 흐름**을 타도록 처리했다.

이를 통해:

- 사용자 입력 흐름을 자연스럽게 유지
- 클릭/엔터에 따른 예외 분기 제거
- 이벤트 처리 구조 단순화

---

### 정리

이번 예외처리 작업은 “모든 예외를 다 처리했다”는 나열이 아니라,

- 어떤 실수가 실제로 자주 발생하는지 생각하고
- 그 실수를 어느 단계에서 막는 게 가장 합리적인지

를 기준으로 설계했다.

결과적으로 **기능은 그대로 유지하면서, 사용자 경험과 서비스 안정성을 함께 보강한 작업**이었다.
